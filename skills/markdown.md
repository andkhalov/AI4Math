# Markdown — паттерны и формат

## Когда Markdown, когда LaTeX

- **Markdown** — README, документация, diary.md, обзоры литературы,
  отчёты экспериментов, notebook-like заметки, GitHub/GitLab артефакты.
- **LaTeX** — см. `load_skill("latex")`. Формальные статьи, диссертации,
  книги.
- Если пользователь не уточнил — выбери формат по контексту: документация
  / заметки → Markdown; статья / диссертация → LaTeX.
- **Не смешивай** LaTeX-специфичные команды в Markdown без крайней нужды.
  Если нужна формула в MD — используй обычный GFM math (`$...$`,
  `$$...$$` для display) или inline-code для небольших вещей.

## GitHub-Flavored Markdown — базовые элементы

```markdown
# Заголовок 1
## Заголовок 2 (максимум 3 уровня)

Обычный параграф. Перенос строки — **две пробела в конце** или пустая
строка для нового параграфа.

**bold** — только для критических терминов, не для выделения каждого слова.
*italic* — для названий, иностранных слов, эмфазы.
`inline code` — команды, имена файлов, идентификаторы.

- Список
- с несколькими
- элементами

1. Нумерованный
2. список
3. (если порядок важен)

> Цитата. Используется для чужих слов или важных замечаний.

[Ссылка](https://example.com) — не пиши raw URL в тексте если можно
дать человекочитаемую подпись.

![alt text](path/to/image.png) — всегда пиши alt text.
```

## Код-блоки

Всегда указывай язык после ` ``` ` — это включает подсветку на GitHub:

    ```python
    def foo():
        return 42
    ```

    ```bash
    git clone https://github.com/user/repo.git
    cd repo && ./setup.sh
    ```

    ```lean
    example : 1 + 1 = 2 := by norm_num
    ```

Для **inline** — одинарные backticks: `path/to/file.py:42`, `variable_name`.

## Таблицы

```markdown
| Column A | Column B | Column C |
|----------|:--------:|---------:|
| left     |  center  |    right |
| cell     |   cell   |     cell |
```

Выравнивание через `:---`, `:---:`, `---:`. GitHub рендерит нормально.

## Ссылки на файлы (в IDE-friendly формате)

Когда упоминаешь файл проекта, используй **кликабельную ссылку** в формате
`[file.py:42](path/file.py#L42)` или `path/file.py:42` — современные IDE
умеют переходить по таким ссылкам одним кликом.

## Математика в MD

GitHub с 2022 рендерит LaTeX math через KaTeX:

```markdown
Inline: $x^2 + y^2 = z^2$

Display:
$$
\int_0^\infty e^{-x^2}\,dx = \frac{\sqrt{\pi}}{2}
$$
```

Ограничения: не все LaTeX-пакеты поддерживаются, сложные environments
(`align`, theorem, proof) работают через `\begin{aligned}...\end{aligned}`
внутри display math. Для серьёзной математики всё равно лучше `.tex`.

## Структура документа

Минимальная сетка:

```markdown
# Название

Одно-два предложения summary.

## Motivation / Context

Зачем это нужно.

## Main content

Разбей на 2-4 секции уровня `##`.

## Usage / Examples

Копи-пейст готовые примеры.

## References

[1] Author, Year. [Title](url). Journal.
```

## Frontmatter (YAML)

Если файл читается каким-то инструментом (Goose skills, Jekyll, mkdocs),
добавляй YAML frontmatter в начале:

```markdown
---
name: skill-name
description: Краткое описание
---

# Contents here
```

Для обычной документации frontmatter не нужен.

## Anti-patterns (не делай)

- **Три+ уровней заголовков** (`####`, `#####`) — реструктурируй вместо этого.
- **Bold для каждой фразы** — теряется смысл выделения.
- **Code блок без языка** — нет подсветки, хуже читается.
- **Длинные ссылки в тексте** (`подробнее здесь https://very.long.url/...`)
  — оборачивай в `[текст](url)`.
- **Таблицы из 20 колонок** — лучше код-блок с фиксированной разметкой
  или несколько таблиц поменьше.
- **HTML-теги** в MD (`<br>`, `<div>`) без необходимости. GFM почти всегда
  достаточен.
- **Emoji в заголовках** (`## 🚀 Install`) — визуальный шум, плохо работает
  в терминалах.

## При создании README

1. Первая строка — `# Project Name`
2. Один-два абзаца — что это и зачем
3. Секция "Quick start" с одной-двумя копи-пейст командами
4. Секция "Usage / Examples"
5. Секция "Docs" / "Architecture" / "FAQ" по необходимости
6. Секция "License" в конце

**Не создавай README без явного запроса пользователя.** Особенно не пиши
`README.md` в проекте который только стартовал — это часто антипаттерн.
