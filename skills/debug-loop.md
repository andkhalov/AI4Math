# Debug loop — дисциплина выполнения собственного кода

## Правило №1: написанный код должен быть выполнен

Код который ты написал и не запустил — это **черновик**, не результат. Не
отдавай пользователю Python/shell/Lean код, которого ты сам не видел в
работе в текущей сессии.

Исключения (когда можно не запускать):

- Код требует внешних API-ключей/данных которых нет в сессии (но скажи об
  этом явно: «не могу проверить, не хватает X»)
- Код для окружения, отличного от текущего (iOS, Windows GUI, embedded)
- Одно-строчный тривиальный ответ где запуск добавит шум
- Снипет как пример документации, явно помеченный `# example only`

Во всех остальных случаях — **цикл write → run → observe → fix**.

## Шаги цикла

1. **Формулировка гипотезы**. Что именно ты проверяешь? «Функция `parse_date`
   парсит ISO-8601 с timezone» — это гипотеза. «Делаем парсер» — нет.

2. **Минимальный эксперимент**. Напиши самый короткий код который проверяет
   гипотезу. Не добавляй лишние фичи, обработку ошибок, логгирование —
   только то что нужно для проверки.

3. **Запуск**. `python3 test.py`, `pytest test.py`, `bash script.sh`,
   `lean_check("example : ...")`. Что угодно с явным выводом.

4. **Наблюдение**. Читай ВЕСЬ stdout и stderr. Ищи:
   - Traceback / error message → надо понять причину
   - Warning → запомни, может быть релевантно
   - Unexpected output → гипотеза была неправильной
   - Ничего не выведено → хорошо, но проверь что код реально выполнился
     (а не был закоммечен / обрушился в `--help`)

5. **Фиксация или пересмотр гипотезы**. Ошибка → fix в коде, возврат к шагу 3.
   Гипотеза была неверной → переформулируй, возврат к шагу 1.

6. **Выход из цикла** при одном из:
   - Код работает как ожидалось → задача решена
   - 5 итераций без прогресса → остановись, опиши проблему пользователю,
     сформулируй что ты пробовал и что всё ещё не ясно
   - Фундаментальное препятствие (нет инструмента, нет доступа) → сообщи
     пользователю явно

## Что считается «запустить»

- Python: `python3 file.py`, `python3 -c "..."`, `pytest`, `jupyter execute`
- Shell: `bash script.sh`, `./script.sh`
- Lean: `lean_check(code)` — см. `load_skill("lean")`
- Компиляция: `gcc/clang/rustc/go build` — компиляция без выполнения уже
  даёт сигнал о синтаксисе
- LaTeX: `pdflatex` — см. `load_skill("latex")`
- Тесты: `pytest`, `go test`, `cargo test` — это тоже выполнение

Не считается выполнением:

- Визуальная проверка кода в редакторе
- `cat file.py` (это просмотр, не выполнение)
- Синтаксический highlighter в IDE (мы не в IDE)
- `python3 -m py_compile file.py` (это bytecode compile, не запуск)

## Bisect — когда код не работает и не понятно где

Если сложный код падает и ошибка непонятна:

1. **Закомментируй половину** проблемного блока
2. Запусти
3. Если падает — проблема в оставшейся половине; комментируй ещё половину
4. Если проходит — проблема в закомментированной; раскомментируй половину
5. Повторяй пока не найдёшь минимальный failing case
6. Когда нашёл — ищи root cause

Альтернативно — вставляй `print()` / `logger.debug()` / `pdb.set_trace()`
на промежуточные значения и смотри где данные расходятся с ожиданием.

## Reproducing bug до fix'а

Если пользователь сообщает о баге:

1. **Сначала воспроизведи** баг ровно так как описано. Если не можешь
   воспроизвести — спроси уточнение ДО того как пытаться чинить.
2. **Минимизируй** repro: урежь до самого простого кода который всё ещё
   показывает проблему.
3. **Только тогда** пиши fix. Проверь что минимизированный repro теперь
   работает.
4. **Убедись что полный сценарий** (не только repro) теперь работает.

Чинить баг которого ты не видел — путь к иллюзорным fix'ам и
никогда-закрытой issue.

## Root cause vs symptom

Если ошибка — «падает с NoneType has no attribute 'foo'», есть два уровня:

- **Symptom fix**: `if x is not None: x.foo()` — прячет баг, но не решает
  «почему x оказался None».
- **Root cause fix**: разобраться где x теряет значение, починить ТАМ.

**Всегда предпочитай root cause**, если только time-to-market не критичен
и symptom fix явно обозначен как временный (`# TODO: investigate`).

## Частые грабли исполнения

- **Stale caches**: `.pyc`, `__pycache__`, `.lake/`, `node_modules` — если
  ведут себя странно, чисти.
- **Wrong Python**: `python` vs `python3` vs `.venv/bin/python`. Проверяй
  `which python3` и `python3 --version` в начале сессии.
- **Stale environment**: переменные окружения из предыдущего запуска. `env |
  grep <prefix>`.
- **File not saved**: написал в памяти IDE, забыл `:w`. Проверь `git status`
  или `stat file.py`.
- **Running from wrong directory**: относительные пути. `pwd` → `cd`.

## Минимум прозрачности пользователю

После цикла debug'а, в финальном ответе укажи:

- Что ты проверял (1 фраза)
- Что выяснилось (1-2 фразы)
- Если проблема исправлена — где именно (file:line)
- Если проблема НЕ исправлена — что остаётся непонятным

Не пиши эссе про свои размышления — пользователь прочитает код и diff сам.
