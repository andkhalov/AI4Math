# Python — паттерны и дисциплина выполнения

## Закрытый цикл выполнения (обязательно)

Любой написанный Python-код должен быть **запущен** в той же сессии. Не отдавай
пользователю код, которого ты сам не видел в работе. Если запустить нельзя
(нет тестовых данных, нет окружения) — скажи об этом явно.

Цикл:

1. Прочитай требования, уточни неясное короткой фразой (не длинным эссе).
2. Напиши минимальный вариант в файл через `text_editor`.
3. Запусти: `shell(python3 <file>)` или `shell(python3 -m pytest <file>)`.
4. Прочитай вывод. Есть ошибка — исправь в том же файле, запусти снова.
5. Максимум 5 итераций. После пятой — остановись, сформулируй гипотезу
   о корневой причине, спроси пользователя.

## Venv и зависимости

- **Сначала** проверь наличие `.venv/` в текущем проекте. Если есть — используй
  `.venv/bin/python` и `.venv/bin/pip`, не системный python.
- Если venv'а нет и ты ставишь сторонние пакеты — создай его:
  `python3 -m venv .venv && .venv/bin/pip install <pkg>`.
- **Не ставь пакеты в системный Python** (`sudo pip install`, `pip install --user`
  без venv). На современных Linux это вернёт `externally-managed-environment`
  — это сигнал создать venv, а не использовать `--break-system-packages`.
- Если пакет уже есть в системных `apt`-репо (numpy, pandas на Debian) — допускается
  `apt install python3-<pkg>` только с явного согласия пользователя.

## Стиль кода

- **Без комментариев про WHAT** (`# increment counter` — мусор). Комментарий
  только про WHY, если это не очевидно из имени переменной.
- **Type hints** на публичные функции и классы. Внутренние helper'ы — опционально.
- **f-strings** вместо `%` и `.format()`.
- **pathlib.Path** вместо `os.path.join`.
- **`logging`** вместо `print` для серверного кода. `print` нормально для
  скриптов и CLI.
- **Ранние возвраты** (guard clauses) вместо глубоких вложений.
- **Явные исключения** (`ValueError`, `FileNotFoundError`), не голый `Exception`.

## Error handling

- **Не глотай исключения**. `try/except` без действия (или только с `pass`)
  — почти всегда баг. Если действительно нужно игнорировать — комментарий
  с пояснением почему.
- **Не делай catchall** `except Exception as e: print(e)`. Ловим конкретные
  типы. Catchall только на верхнем уровне скрипта/обработчика.
- **Failure boundary** — граница между пользовательским вводом и внутренним
  кодом. Внутри доверяй инвариантам, на границе — валидируй.

## Тестирование

- Предпочитай **`pytest`** встроенному `unittest`.
- Тесты — отдельный файл `tests/test_<module>.py` или `<module>_test.py`.
- Один тест — одно утверждение (`assert`), осмысленное сообщение при
  падении.
- **Не пиши тесты «для галочки»** (тест который проверяет что `1 == 1`).
- Если пишешь новый код с чистого листа — TDD: сначала один failing тест,
  потом реализация до зелёного.

## Jupyter notebook

- Jupyter notebook — **плохой выбор по умолчанию**. Используй `.py` скрипт
  если можно. Notebook оправдан только для: интерактивный EDA, демонстрация,
  обучение.
- Если пользователь явно попросил notebook (`.ipynb`):
  - **Каждая cell должна выполняться в изоляции от стейта предыдущих**, если
    это не явно sequential ETL. Проверь: после kernel restart и "Run All"
    notebook проходит до конца без ошибок.
  - **Никаких stale variables**. Не оставляй `x = 5` в одной cell и `print(y)`
    в другой если `y` нигде не определено.
  - **Первая cell** — imports. Последняя — пустая или summary.
  - После написания — **запусти notebook через `jupyter nbconvert --to notebook
    --execute <file>.ipynb`** или `jupyter execute <file>.ipynb`. Это эквивалент
    Run All. Если падает — правь.

## Стандартная библиотека сначала

Не тяни зависимость ради 3 строк. `json`, `pathlib`, `subprocess`, `urllib`,
`argparse`, `dataclasses`, `datetime`, `itertools`, `collections`, `functools`,
`re` — покрывают 80% потребностей скриптов.

Исключения, когда зависимость оправдана:

- `requests` — когда `urllib` слишком уродлив
- `pydantic` — когда нужна валидация моделей
- `numpy` / `pandas` — численные данные, только если объёмы реально требуют
- `pytest` — всегда предпочтительнее `unittest`

## Anti-patterns (не делай)

- `from module import *` — непонятно что пришло.
- Изменение глобальных переменных внутри функций без `global`.
- `eval` / `exec` на внешнем вводе.
- Mutable default arguments (`def f(x=[]):`). Используй `x=None` → `if x is None: x = []`.
- Циклы `for i in range(len(items))` когда нужен `for item in items` или `enumerate`.
- `except: pass` без объяснения.
