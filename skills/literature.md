# Обзор литературы — workflow

## Когда применяется

Пользователь просит:
- «Сделай обзор литературы по теме X»
- «Найди статьи про Y и проанализируй»
- «Скачай PDF и расскажи что там»
- «Какие есть основные работы по Z»

## Алгоритм

### 1. Поиск кандидатов

`web_search(query, num_results=10)` с релевантным запросом. Для
peer-reviewed источников добавляй квалификаторы:

- `site:arxiv.org` — arXiv preprints
- `filetype:pdf` — PDF документы
- `site:proceedings.neurips.cc` / `site:aclanthology.org` / etc. — конкретные конференции

Читай результаты: `title / url / snippet`.

### 2. Уточнение у пользователя (важно — не качай всё подряд)

Покажи найденные результаты нумерованным списком и спроси что скачивать:

```
Нашёл 8 работ:
1. <Title> — <short description> — <url>
2. ...
...
Какие скачать для детального анализа? Укажи номера (например "1, 3, 5")
или "все".
```

**Исключения** (можно пропустить этот шаг):
- Пользователь сразу сказал «скачай все» или «скачай первые N»
- Пользователь явно назвал одну конкретную работу
- Найден всего один релевантный результат

### 3. Подготовка папки

```
mkdir -p literature
```

Если папка уже есть — не трогай существующие файлы.

### 4. Скачивание PDF

Для каждой выбранной работы:

```
pdf_download(url="https://arxiv.org/pdf/...", dest_path="literature/")
```

- `dest_path` заканчивающийся на `/` → имя файла берётся из URL.
- **Не используй `web_fetch` для PDF** — он декодирует ответ как текст и
  ломает бинарный контент. `pdf_download` использует binary-safe streaming.
- Если ответ `ERROR: ...` — запиши причину, пропусти этот файл, продолжи
  следующие. Не останавливай весь обзор из-за одного fail'а.
- Если URL не arxiv PDF напрямую (например landing page) — попробуй
  угадать прямую ссылку на PDF (`.../abs/` → `.../pdf/`), или fallback на
  ручной `curl -L`.

### 5. Глубокий анализ каждого PDF

Для каждого скачанного файла:

```
pdf_info(path)                         # метаданные, страницы, размер
pdf_read(path, pages="1-3")            # intro + abstract
pdf_search(path, query="<key term>")   # упоминания термина
```

При необходимости — конкретные секции:

```
pdf_read(path, pages="5-8")            # methods
pdf_read(path, pages="10-12")          # results
```

**Правило**: если работ >5, читай только первые 3 страницы + search по
ключевым терминам. Полное чтение оставь на отдельный запрос пользователя.

**Никогда не выдумывай** содержание. Если `pdf_read` вернул пустоту
(сканированный PDF без OCR) — отметь файл как «требует ручной обработки»,
не фантазируй.

### 6. Отчёт `literature/review.md`

Создай файл через `text_editor`:

```markdown
# Обзор литературы: <тема>

## Список работ

### 1. <Author, Year>. <Title>

- **Файл**: [literature/<fname>.pdf](literature/<fname>.pdf)
- **URL**: <url>
- **Ключевые идеи** (2-3 предложения): ...
- **Методология**: ...
- **Результаты**: ...

### 2. ...

## Связи между работами

<1-2 абзаца>

- Работа #1 опирается на метод X из работы #3 (см. страница 5)
- Работы #2 и #4 оба обсуждают концепцию Y, но расходятся в ...
- Подход #5 критикует ... из #1 на основании ...

**Не фантазируй связи из памяти**. Если реально не видел цитирования в
скачанных PDF — либо найди через `pdf_search(query="author_name_of_other_work")`,
либо пропусти.

## Ключевые термины

- **Term A** — определение (источник: работа #1, стр. 3)
- **Term B** — определение (#3, стр. 7)

## Что осталось прочитать

<Если есть упоминания работ которые не скачаны или не найдены>

- <Author>, <Year>. <Title> — упомянуто в работе #1 (стр. 5), не найдено
  в web_search.
```

Подстраивай структуру под область:

- **Формальная математика**: добавь секцию «Формализация в Lean 4
  (candidate lemmas)» со списком теорем которые можно попробовать
  формализовать через `lean_search_scilib`.
- **ML / NLP**: добавь «Методология» с описанием архитектур, датасетов,
  метрик.
- **Системные работы**: «Архитектура» и «Эксперименты».

### 7. Короткий summary в чат

После того как записал `review.md`, отчитайся пользователю **одним-двумя
абзацами**:

- Сколько работ скачано и проанализировано
- Путь к отчёту (`literature/review.md`)
- 2-3 главных вывода
- Что осталось непонятным / требует следующего шага

**Не дублируй весь отчёт в чат** — пользователь откроет файл если нужно.

## Правила

- **Не выдумывай цитаты и номера страниц**. Используй `pdf_search` для
  поиска конкретного термина и возвращай реальные номера страниц.
- **Не изобретай связи** между работами «по мотивам». Связь = либо
  реальное цитирование одной работы в другой, либо общий термин /
  метод / датасет.
- **Не скачивай всё подряд**. Уточни у пользователя (шаг 2) если только
  он явно не отказался.
- **Peer-reviewed фильтр** — если пользователь просил только рецензируемые
  источники, отметь не-рецензируемые (блоги, GitHub repos, wikipedia) в
  списке как «reference only» и не включай в основной анализ.
- **После записи отчёта** — один `shell(ls -la literature/)` чтобы убедиться
  что все файлы и `review.md` на месте. Это часть execution loop.
